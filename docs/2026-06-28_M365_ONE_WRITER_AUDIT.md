## One-Writer Rule Enforcement

The one-writer rule for M365 is enforced by three layers:

1. **A1 boundary (current):** GAIL OS current autonomy level is A1 — local, no-network. No live GAIL OS Connector execution is permitted. Existing Adam-approved Power Automate proof flows may remain started, but they are not Phase 4 connector execution and do not authorize new agent writes.

2. **Connector registry (Phase 4):** Every write surface will have exactly one `ConnectorProfile` in the GAIL OS Connector Registry. The `prohibited_capabilities` list on each profile explicitly blocks capabilities not granted. The `stop_triggers` list halts execution on any out-of-scope access.

3. **Authority envelope (Phase 4):** Every approved action requires a signed `AuthorityEnvelope` from GAIL OS. The envelope names the specific connector, domain, allowed action types, max risk tier, and stop conditions. Two connectors cannot both hold active envelopes for the same write surface.

## Evidence-Linkage Design

Every M365 write in Phase 4 must produce an `EvidencePacket` that includes:

| Field | Value source |
|---|---|
| `evidence_id` | Generated by GAIL OS on completion |
| `mission_id` | Parent mission that authorized the write |
| `action_id` | The specific action record |
| `envelope_id` | AuthorityEnvelope that governed the write |
| `result` | `success`, `failure`, `stopped`, or `partial` |
| `execution_mode` | `dry-run` or `live` |
| `rollback_note` | Required for any `live` write |
| `outcome_summary` | What was written and to which surface |

The `source-ref.schema.json` (GAIL OS `contracts/json-schema/`) provides the source-of-truth anchor:
- `source_system`: `m365`
- `ref_type`: e.g. `sharepoint-list-item`, `planner-task`, `teams-message`
- `ref_value`: SharePoint item ID, Planner task ID, Teams message ID
